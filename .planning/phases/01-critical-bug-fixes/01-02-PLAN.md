<objective>
修复 BUG-02：`startTaskQueued` 中的 `startTask()` 调用无超时保护，当 SSH 连接卡住（无响应）时，
整个全局 Promise 链（`startQueue`）永久冻结，所有后续任务（包括健康检测触发的自动重启）均无法启动。

所有任务的自动重启机制完全失效，用户无感知地损失直播转推服务。

// AbortSignal.timeout(ms) 在指定毫秒后自动 abort
const signal = AbortSignal.timeout(30000);

// 创建一个在 signal abort 时 reject 的 Promise
function abortPromise(signal) {
  return new Promise((_, reject) => {
    signal.addEventListener('abort', () => {
      reject(new Error(`操作超时（${signal.reason?.message || 'timeout'}）`));
    }, { once: true });
  });
}

// 使用 Promise.race 实现带超时的等待

  <action>
    在 task-manager.js 中实现两步修改：
    
    **步骤 1**：在文件顶部常量区（第 18 行 `let startQueue` 附近）添加超时辅助函数：
    
    在 `let startQueue = Promise.resolve();` 这一行的**上方**添加：
    ```js
    // 返回一个在 signal 触发 abort 时 reject 的 Promise
    // 用于与业务 Promise 竞争，实现强制超时
    function raceAbort(promise, signal) {
      return new Promise((resolve, reject) => {
        // 如果 signal 已经 abort（罕见情况），立即 reject
        if (signal.aborted) {
          reject(new Error(`[队列启动] 操作已超时`));
          return;
        }
        const onAbort = () => reject(new Error(`[队列启动] 操作超时（30s），SSH 无响应`));
        signal.addEventListener('abort', onAbort, { once: true });
        promise.then(
          (val) => { signal.removeEventListener('abort', onAbort); resolve(val); },
          (err) => { signal.removeEventListener('abort', onAbort); reject(err); }
        );
      });
    }
    ```
    
    **步骤 2**：修改 `startTaskQueued` 函数体，用 `raceAbort` 包装 `startTask` 调用：
    
    将现有的：
    ```js
    function startTaskQueued(taskId, userId = null) {
      startQueue = startQueue.then(async () => {
        try {
          await startTask(taskId, userId);
        } catch (e) {
          console.error(`[队列启动] 任务 ${taskId} 失败:`, e.message);
          db.prepare(`UPDATE tasks SET status='error', remote_pid=NULL WHERE id=? ${userId ? 'AND user_id=?' : ''}`)
            .run(...(userId ? [taskId, userId] : [taskId]));
        }
        const delay = parseInt(getSetting('start_delay', userId || undefined) || '5') * 1000;
        await new Promise(r => setTimeout(r, delay));
      });
      return startQueue;
    }
    ```
    
    修改为：
    ```js
    const START_TASK_TIMEOUT_MS = 30 * 1000; // SSH 无响应时的最大等待时间
    
    function startTaskQueued(taskId, userId = null) {
      startQueue = startQueue.then(async () => {
        const signal = AbortSignal.timeout(START_TASK_TIMEOUT_MS);
        try {
          await raceAbort(startTask(taskId, userId), signal);
        } catch (e) {
          console.error(`[队列启动] 任务 ${taskId} 失败:`, e.message);
          db.prepare(`UPDATE tasks SET status='error', remote_pid=NULL WHERE id=? ${userId ? 'AND user_id=?' : ''}`)
            .run(...(userId ? [taskId, userId] : [taskId]));
        }
        const delay = parseInt(getSetting('start_delay', userId || undefined) || '5') * 1000;
        await new Promise(r => setTimeout(r, delay));
      });
      return startQueue;
    }
    ```
    
    **关键点说明**：
    - `AbortSignal.timeout(ms)` 在 Node.js 17.3+ 可用（项目使用 Node 24，无兼容问题）
    - `raceAbort` 在 Promise 完成或超时后都会清理事件监听器，避免内存泄漏
    - 超时后 catch 块捕获错误，任务被标记为 `error` 状态，队列继续推进
    - `startTask` 本身仍然在后台运行直到 SSH 超时（`readyTimeout: 15000`），
      但不再阻塞队列——这是可接受的权衡（后台泄漏的连接会在 ssh.js 的连接池中最终超时）

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| VPS SSH | 远程 SSH 命令执行，可能无响应 |
| Promise 队列 | 全局单链，一个任务卡住影响所有任务 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Denial of Service | startQueue 全局链 | mitigate | AbortSignal.timeout(30s) 保证每个任务最多占用 30s |
| T-02-02 | Tampering | 超时后 startTask 后台继续运行 | accept | startTask 的 SSH 连接最终会被 readyTimeout 或操作系统 TCP 超时关闭 |
| T-02-03 | Denial of Service | raceAbort 事件监听器泄漏 | mitigate | 在 resolve/reject 时都调用 removeEventListener 清理监听器 |
</threat_model>
